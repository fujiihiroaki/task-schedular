## Context

ペースタグは `lead` タグの簡易版として導入された機能であり、`pace:slow|normal|fast` の3段階でリードタイム（締切から逆算した着手日までの日数）を設定できる。現在の実装は以下の通り:

- `MarkdownTaskParser` が `pace:<val>` を正規表現でパースし、`TaskItem.Pace` に文字列として保存
- `Inference.ApplyAutoStart()` で `_PaceToLead()` により日数に変換し、`Lead` を設定
- `MarkdownRenderer` が `pace:{value}` 形式でそのまま出力

今回の変更で、新しいペースレベル（`nonstop`, `rapid`）の追加、既存レベルの日数変更、およびカスタム日数指定（`pace:<name>=<days>`）を実現する。

## Goals / Non-Goals

**Goals:**
- `nonstop`（1日）と `rapid`（7日）を新しいペースレベルとして追加する
- 既存レベルの日数を `slow=90`, `normal=30`, `fast=14` に変更する
- `pace:<name>=<days>` 形式でカスタム日数を指定可能にする
- 不正なペース値に対するバリデーションとエラーメッセージ出力を追加する
- 出力時のペースタグを正規化する（小文字統一、カスタム日数の保持）

**Non-Goals:**
- 既存ユーザーへの移行ツールやマイグレーション支援
- ペースタグのセクション単位でのデフォルト設定
- `start` / `lead` / `pace` の優先順位の変更（現行仕様を維持）

## Decisions

### Decision 1: `TaskItem.Pace` プロパティの型を `string?` のまま維持する

**選択肢:**
- A) `string?` のまま維持し、検証済み・正規化済みの値のみ保持する
- B) 専用の `PaceInfo` クラスを導入して名前と日数を分離管理する

**決定:** A を採用。

**理由:** 現行構造との互換性を維持しつつ、パーサー段階で値を正規化することで設計の複雑化を避けられるため。

### Decision 2: `pace` の検証と正規化はパーサー段階で実施する

**現在の正規表現:**
```
\b(?<key>due|start|lead|pace|p|est|tag)\s*:\s*(?<val>[^\s]+)\b
```

**方針:** `MetaToken` 正規表現は変更しない。`case "pace"` で `TryNormalizePace(raw, out normalized, out error)` を呼び出し、妥当な値のみ `TaskItem.Pace` に設定する。

**正規化ルール:**
- ペース名は小文字化する（`ToLowerInvariant()`）
- `pace:<name>` の `<name>` は `[a-z][a-z0-9_-]*` を許可し、既定ペース名（`nonstop`, `rapid`, `fast`, `normal`, `slow`）のみ有効とする
- `pace:<name>=<days>` の `<days>` は整数として解釈し、`1..2048` の範囲のみ有効とする
- カスタム日数は整数表記に正規化する（例: `007` → `7`）
- 同一タスクで複数の `pace` がある場合、後で読んだ**有効な**指定を採用する（無効値は上書きに使わない）

**理由:** `due` の有無に依存せず常に検証できるため、「不正値は無視」の要件を確実に満たせる。

### Decision 3: `_PaceToLead()` は正規化済み `pace` から日数へ変換する責務に限定する

**方針:** `_PaceToLead()` は検証済みの `TaskItem.Pace` を入力として扱い、日数変換のみ行う。

1. `pace` に `=` がある場合は右辺の整数を `TimeSpan.FromDays()` に変換する
2. `=` がない場合は既定ペース名から日数へマップする（`nonstop=1`, `rapid=7`, `fast=14`, `normal=30`, `slow=90`）
3. 変換不可時は `null` を返す（防御的実装）

**理由:** 検証と変換の責務を分離し、エラー処理の重複を避けるため。

### Decision 4: バリデーションエラーは詳細付きで `Console.Error` に出力し、当該値を無視する

**方針:** 不正な `pace` 値（未知の名前、空名、範囲外、非整数）を検出した場合:
- `Console.Error.WriteLine()` で1行警告を出力する
- フォーマットは「値」「理由」「タスク識別子」を含める  
  例: `[pace] invalid value 'FAST=0' (reason: days out of range 1..2048, id:xxxx, title:タスク名)`
- その `pace` は設定しない（既存の `lead` 推論へフォールバック）
- 処理は継続する

**理由:** ユーザーが修正箇所を即時特定できるようにするため。

### Decision 5: レンダリングは正規化済み値を出力する

**方針:**
- `TaskItem.Pace` は正規化済み値のみ保持される前提とする
- 既定ペースは `pace:<name>` 形式で出力する
- カスタムは `pace:<name>=<days>` 形式で出力する（`<days>` は正規化済み整数）

**理由:** 出力フォーマットの一貫性を保証するため。

### Decision 6: README を仕様と同期して更新する

**方針:**
- `pace` の既定日数説明を `slow=90`, `normal=30`, `fast=14` に更新する
- 追加ペース `nonstop=1`, `rapid=7` を追記する
- `pace:<name>=<days>` 形式とバリデーション範囲（`1..2048`）を追記する

**理由:** proposal で合意した公開仕様とユーザー向け説明を一致させるため。

## Risks / Trade-offs

- **[既存ペース日数の変更]** → `slow` が 120→90 日、`normal` が 90→30 日、`fast` が 30→14 日に変更されるため、既存ユーザーのタスクのリードタイムが変わる。proposal で「移行考慮は行わない」と決定済み。
- **[カスタム日数の上限]** → 2048 日（約5.6年）を上限としているが、極端に大きな値でも動作はする。上限を超えた場合は無視される。
- **[不正値の無視]** → 不正なペース値は警告のみで処理を継続するため、ユーザーがタイポに気づきにくい可能性がある。→ `Console.Error` への出力で最低限の通知を行う。
