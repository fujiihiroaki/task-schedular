## Context

現在の入力ファイルはタスクごとに `due:` を明示指定するか、セクション見出しの日付（`SectionDate`）をフォールバックとして使用している。ファイルレベルのメタデータ（frontmatter 等）は存在しない。

「今期」のような業務サイクルに沿った期間管理を行うには、ファイルに期間を定義し、特定セクションのタスクに期末日を自動付与する仕組みが必要である。

## Goals / Non-Goals

**Goals:**
- 入力ファイルに「今期」の期間（終了日）を宣言できるようにする
- 「今期の」で始まるセクション内の未完了タスクに期末日を `due` として自動設定する
- 既存の `due:` 指定を尊重し、上書きしない
- 後方互換性を維持する（期間未定義のファイルは従来通り動作する）

**Non-Goals:**
- 複数期間（来期、前期など）の管理
- 期間の自動計算（会計年度ベースの推定など）
- カレンダー統合や外部サービス連携

## Decisions

### Decision 1: 期間定義の構文

メタデータ行として `period: YYYY-MM-DD .. YYYY-MM-DD` 形式をファイル先頭に記述する。

```markdown
period: 2026-01-01 .. 2026-03-31

# 今期のバックログ
- [ ] 資料整理
- [ ] レビュー対応
```

**理由:** 既存のメタデータキー（`due:`, `start:` 等）と同じ `key: value` パターンに統一できる。frontmatter（YAML）は既存パーサーに YAML 依存を持ち込むため避ける。

**代替案:**
- YAML frontmatter (`---` 区切り): YAML パーサー依存が増える。既存の `key: value` パターンと一貫性がない
- コメント形式 (`<!-- period: ... -->`): 既に ID 埋め込みで使用しているが、ユーザーが直接編集するメタデータには不向き

### Decision 2: 期間の適用ルール

`period:` が定義されている場合、セクション名が「今期の」で始まるセクション内のタスクに対して、期間の終了日を `due` として設定する。

**適用条件:**
1. `period:` がファイルに定義されている
2. タスクが「今期の」で始まるセクション内にある
3. タスクに明示的な `due:` が指定されていない

**理由:** 既存の `item.Due ??= item.SectionDate` パターンと同様のフォールバック方式を採用し、明示指定を常に優先する。

### Decision 3: 実装箇所

`MarkdownTaskParser.cs` の `Parse()` メソッド内で `period:` 行を検出し、期間終了日を保持する。`Inference.cs` でセクション名判定と `due` 自動設定を行う。

**理由:** パース（構文解析）と推論（ビジネスロジック）の責務分離を維持する。既存のパイプライン（Parse → Inference → Rank → Render）に自然に統合できる。

### Decision 4: period 行の配置制約

`period:` 行はファイル内の最初のタスクまたは見出しより前に記述する。複数定義された場合は最後の定義が有効となる。

**理由:** ファイルレベルのメタデータとして扱うため、ヘッダー領域に配置するのが自然。複数定義時に最後を採用するのは既存メタデータの上書きパターンと一致する。

## Risks / Trade-offs

- **[セクション名の部分一致]** 「今期の」で始まるかどうかの判定は文字列前方一致で行うため、意図しないセクション名がマッチする可能性がある → ユーザーが命名規則を理解していれば低リスク。ドキュメントで明記する
- **[単一期間の制約]** 1ファイルに1つの `period:` のみ対応するため、複数プロジェクトで異なる期間を持つ場合に対応できない → 将来的に複数期間対応を検討可能。現時点では単一で十分
- **[後方互換性]** `period:` 行が存在しないファイルでは一切の動作変更なし → リスクなし
